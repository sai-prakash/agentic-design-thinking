# How-Might-We Question Patterns

How-Might-We (HMW) questions reframe problems as opportunities for design. They are broad enough to allow creative solutions but narrow enough to provide useful constraints. A good HMW question opens the door to ideation without prescribing a specific answer.

## From POV to HMW

Start with a Point-of-View statement:

> [User] needs [need] because [insight].

Then reframe it as: **"How might we [opportunity]?"**

Example POV: "Sarah, a project manager, needs a way to see overdue tasks across all projects at a glance because she spends 20 minutes each morning manually compiling status from three separate boards."

Direct HMW: "How might we give project managers instant visibility into overdue tasks across all their projects?"

## Pattern 1: Amplify the Positive

Take something that is already working and make it stronger.

- "How might we make the relief Sarah feels when tasks are completed on time a more frequent experience?"
- "How might we build on the habit of morning board-checking to deliver more value in that moment?"
- "How might we amplify the satisfaction users feel when they complete a complex form successfully?"
- "How might we extend the trust users have in our payment system to other parts of the product?"
- "How might we make the onboarding momentum carry into the first week of regular use?"

## Pattern 2: Remove the Negative

Eliminate or reduce pain points, frustrations, and obstacles.

- "How might we eliminate the need for Sarah to manually copy overdue tasks into a spreadsheet?"
- "How might we reduce the anxiety project managers feel before stakeholder meetings?"
- "How might we remove the friction of switching between multiple project boards?"
- "How might we prevent users from receiving notifications that are not relevant to them?"
- "How might we stop error messages from blocking users who made a simple typo?"

## Pattern 3: Explore the Opposite

Challenge assumptions by flipping the problem around.

- "How might we make the task board update itself so Sarah never has to ask for status?"
- "How might we let overdue tasks find the project manager instead of the other way around?"
- "How might we design a system where stakeholders pull their own status reports?"
- "How might we make the notification system so precise that users want more notifications, not fewer?"
- "How might we make form errors feel like helpful guidance rather than punishment?"

## Pattern 4: Question Assumptions

Identify and challenge hidden assumptions in the problem statement.

- "How might we help Sarah if she did not need a centralized board at all?"
- "How might we support project management if team members never explicitly update task status?"
- "How might we serve users who do not read error messages?"
- "How might we design for the case where the user's goal changes mid-task?"
- "How might we provide navigation if users do not think in terms of pages and menus?"

## Pattern 5: Break It into Parts

Decompose a complex challenge into smaller, more tractable pieces.

- "How might we help Sarah identify which tasks are overdue?" (detection)
- "How might we help Sarah understand why tasks are overdue?" (diagnosis)
- "How might we help Sarah take action on overdue tasks?" (resolution)
- "How might we help Sarah prevent tasks from becoming overdue?" (prevention)
- "How might we help Sarah communicate overdue status to stakeholders?" (reporting)

## Quality Checklist for HMW Questions

A good HMW question should pass these tests:

1. **Not too broad**: "How might we improve the app?" is too vague to guide ideation.
2. **Not too narrow**: "How might we add a red badge to the overdue tab?" prescribes a solution.
3. **User-centered**: References a real user need, not a business metric.
4. **Generative**: Inspires multiple possible solutions, not just one obvious answer.
5. **Grounded in research**: Based on observed behavior or stated needs, not assumptions.

## Using HMW in the Define Stage

Generate 5-8 HMW questions per POV statement. Use at least 3 different patterns to ensure diversity. Prioritize 2-3 questions that are most generative and best aligned with the project scope. These become the creative brief for the Ideate stage.
